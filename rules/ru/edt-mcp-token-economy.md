# Экономия токенов при работе с EDT-MCP

> Каждый запрос к MCP-серверу — это токены. На конфигурациях БСП один `read_module_source` без диапазона может стоить десятки тысяч токенов. Эти правила экономят 5-50× в зависимости от задачи.

## Принципы

1. **Сначала сводка, потом детали.** Иерархия «дешёвый → дорогой» (всегда начинай с левого):
   - `get_problem_summary` → `get_project_errors` (с фильтрами)
   - `get_module_structure` → `read_method_source` → `read_module_source`
   - `search_in_code outputMode=count` → `outputMode=files` → `outputMode=full`
   - `get_metadata_objects` → `get_metadata_details`

2. **Читай метод, а не модуль.** `read_method_source` вместо `read_module_source`. Для модулей БСП (6000+ строк) экономия в десятки раз. К `read_module_source` обращайся только когда метод завязан на `#Область` или модульные переменные.

3. **Используй фильтры, которые есть в инструменте.** Не запрашивай всё с намерением «отфильтрую сам» — это худший антипаттерн по токенам. См. таблицу ниже: почти у каждого инструмента есть `nameFilter` / `metadataType` / `severity` / `objects` / `contains` / `checkId` / `fileMask`.

4. **Не повторяй запросы.** Если ответ есть в контексте сессии — не вызывай снова. Если боишься «забыть» — сохрани результат во временный файл проекта (`.cache/edt-mcp/…` или аналог) и читай оттуда. Это особенно актуально для крупных карт: `get_module_structure`, `list_modules`, `get_metadata_objects`.

5. **Батчи там, где принимаются.** `get_metadata_details` принимает массив `objectFqns` — собирай батч, не вызывай по одному объекту.

6. **Останавливайся на достаточном.** Если уже знаешь имя метода и нужно прочитать его тело — `read_method_source`, без предварительного `get_module_structure`. Не делай «обзорные» вызовы без явной нужды.

## Конкретные правила по инструментам

### `get_problem_summary` (дешёвый)

- Один вызов — счётчики по проектам и severity. **Всегда первым** перед `get_project_errors`. Сразу видно, имеет ли смысл копать дальше.

### `get_project_errors` (дорогой)

- `limit` — по умолчанию 100, максимум 1000. **Не повышай без нужды.**
- `severity` — `ERRORS`, `BLOCKER`, `CRITICAL`, `MAJOR`, `MINOR`, `TRIVIAL`. Чаще всего интересны первые три.
- `objects` — массив FQN (`['Document.SalesOrder', 'Catalog.Products']`). Сужает на порядки.
- `checkId` — подстрока кода проверки (например, `ql-temp-table-index`). Полезно когда ищешь конкретный класс ошибок.
- На свежей конфигурации без правок могут быть тысячи ошибок — **не вытягивай всё**, фильтруй.

### `search_in_code`

- Параметр количества называется **`maxResults`** (не `limit`): default 100, max 500.
- `outputMode`: `count` (только число), `files` (файлы + счётчики), `full` (с контекстом, default). Используй в этом порядке.
- `metadataType` — точнее, чем `fileMask`. Если ищешь только в общих модулях — `metadataType: commonModules`.
- `contextLines` — default 2, max 5. Поднимать без нужды не стоит.
- Это **замена `Grep` по `.bsl` файлам**. Не читай весь модуль ради одной строки.

### `get_module_structure` (дешёвый)

- Возвращает только сигнатуры процедур/функций, регионы, параметры — без тел.
- **Перед** `read_module_source`/`read_method_source` — для незнакомого модуля.
- `includeVariables: false` (default) — модульные переменные обычно не нужны для карты.
- `includeComments: false` (default) — комментарии тоже раздувают вывод.

### `read_method_source` (предпочтительный для чтения BSL)

- Точечное чтение одного метода по имени. Если метода нет — вернёт **список доступных** имён (можно повторно вызвать с правильным).
- В ответе есть `region` — сразу видно, в какой `#Область` он лежит.

### `read_module_source` (дорогой)

- Используй **только** если: метод завязан на `#Область`, нужно увидеть директивы модуля, либо нужно соседство нескольких методов.
- Всегда передавай `startLine`/`endLine`, если можешь оценить диапазон.
- Параметр `maxLines` настраивается в `Window → Preferences → MCP Server → Tools` (default 500, максимум 50000). При обрезании ответ содержит `truncated: true` и `nextStartLine` — продолжай чтение **только если действительно нужно**, не дочитывай весь модуль автоматически.

### `list_modules`

- В большой конфигурации — тысячи модулей. Всегда фильтруй.
- `metadataType` (например, `documents`) или `objectName` (например, `Products`) — резко сужают.
- `nameFilter` — подстрока пути.
- `limit` default 200.

### `get_metadata_objects`

- **Всегда** ставь `metadataType` (а не `all`), если знаешь область интереса.
- **Всегда** ставь `nameFilter`, если ищешь по подстроке.
- `limit` default 100, max 1000.

### `get_metadata_details`

- Принимает массив `objectFqns` — собирай **батч** (`['Catalog.X', 'Document.Y']`), не вызывай по одному.
- `full: false` (default) — обзор. **Достаточно в 90% случаев.**
- `full: true` — только когда нужны все свойства (реквизиты, формы, реквизиты табличных частей и т. д.). Раздувает ответ в разы.

### `find_references`

- Default 100, максимум 500. Поднимай только для рефакторинга, требующего полного охвата.
- Поддерживает **только top-level** объекты (`Catalog.X`, `Document.Y`, `CommonModule.Z`). Для вложенных вернёт ошибку — переходи сразу к `rename_metadata_object`/`delete_metadata`.

### `get_content_assist`

- `extendedDocumentation: true` — **только** когда правда нужно полное описание. Иначе сильно раздувает ответ.
- `contains` — фильтр по подстроке (через запятую: `'Insert,Add'`). Режет ответ до релевантного.
- `offset` — есть пагинация. Если нужно «второй пачкой» — `offset: 100, limit: 100`.
- `limit` default 100, max 1000. На большинство задач хватает 20-50.

### `get_form_layout_snapshot`

- `mode: compact` (default) — только видимые элементы с положительными bounds и ключевыми display-свойствами. **Используй по умолчанию.**
- `mode: full` — все узлы и все non-containment свойства. Это десятки тысяч токенов на сложной форме. Включай **только** если compact не даёт нужной детали.

## Кэширование в файлы проекта

Если в течение сессии многократно нужны **одни и те же** большие срезы (карта проекта, список модулей, FQN-каталог) — сохрани результат **один раз** во временный markdown в проекте и работай с ним:

- Подходит для: `get_module_structure` больших модулей, `list_modules`, `get_metadata_objects` для конфигурации, `find_references` для опорных объектов рефакторинга.
- Не подходит для: `get_project_errors` (быстро устаревает после правок), результатов отладки, `get_form_layout_snapshot` (зависит от состояния редактора).
- Файл клади в gitignored путь (например, `.cache/edt-mcp/<имя>.md`). Не коммить такие срезы.

## Антипаттерны

- `read_module_source` без `startLine`/`endLine` для модуля БСП — съест бюджет одного вызова.
- `search_in_code outputMode=full` сразу, без оценки масштаба через `count` — может вернуть мегабайты.
- `get_metadata_details full=true` для обзорного просмотра — почти никогда не нужно.
- `get_content_assist extendedDocumentation=true` для рутинной проверки имени метода.
- Цикл `get_metadata_details` по одному FQN вместо одного батч-вызова.
- Повторный запрос за тем же `get_module_structure` в течение сессии вместо сохранения результата.
- `get_project_errors` без `severity` и `objects` на конфигурации с тысячами проблем.
- Использование клиентских `Read`/`Grep` по `.bsl` файлам вместо `read_method_source`/`search_in_code` (см. `INDEX.md`, секция NEVER).

## Чек-лист «перед запросом»

- [ ] Знаю ли я уже ответ из контекста сессии или из кэш-файла?
- [ ] Есть ли «дешёвый» инструмент того же класса, который даст 80% ответа? (`*_summary`, `*_structure`, `outputMode: count`)
- [ ] Применил ли я все доступные фильтры (`metadataType`, `nameFilter`, `severity`, `objects`, `contains`, `fileMask`)?
- [ ] Нужен ли мне полный объект, или достаточно метода/сводки/диапазона строк?
- [ ] Это второй вызов с теми же параметрами за сессию — почему?
