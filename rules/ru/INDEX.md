## Приоритет конфликтов

```
текущий запрос пользователя > CLAUDE.md (проектные правила) > rules/ > внутренняя документация проекта
```

Самый специфичный файл побеждает более общий. Например, `bsl-ssl.md` уточняет `bsl-coding-style.md`.

**Если текущий запрос пользователя противоречит правилу из этой папки — следуй пользователю.** Не молчи и не отказывай. Одним предложением скажи, какое правило ты сейчас обходишь и почему (по запросу). Если отклонение опасное (затрагивает безопасность, может уничтожить данные) — сначала переспроси, потом действуй.

## NEVER (никогда)

- Для чтения и правки `.bsl` файлов — только инструменты MCP EDT (`read_method_source`, `read_module_source`, `write_module_source`). Прямое чтение/правка `.bsl` файловыми инструментами клиента (Read/Edit/Write у Claude, аналоги в Cursor/Copilot и т. п.) ломают контекст и впустую тратят токены
- Не вызывать `write_module_source` с `mode: replace` для одного метода — это перепишет **весь модуль**. См. `edt-mcp-write-safety.md`
- Не использовать плейсхолдер-UUID (`00000000-...`, `a1b2c3d4-...`) в `.mdo` файлах — только криптографически случайные v4. См. `edt-metadata.md`
- Не делать git разрушительные операции (`reset --hard`, `push --force`, `checkout .`) без явного запроса пользователя
- Не коммитить секреты (`.env`, токены, ключи)

## ALWAYS (всегда)

- Перед правкой объекта BSL — прочитай метод через `read_method_source`, не загружай модуль целиком
- Перед `rename_metadata_object` / `delete_metadata` — сначала вызов **без** `confirm` (preview всех изменений), потом тот же вызов с `confirm: true` для применения
- После любой записи кода — проверь ошибки через `get_project_errors` или `get_problem_summary`
- Перед написанием нового кода — найди похожие паттерны в проекте через `search_in_code` и следуй им
- При сомнениях в сигнатуре платформы 1С — `get_platform_documentation`, не угадывай
